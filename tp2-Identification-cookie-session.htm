Partie A: Identification par le protocole HTTP

Le protocole http prévoit un champ correspondant à une demande d'identification : WWW-Authenticate.

Le client, lors d'une demande d'envoi de page (protégée) reçoit une trame en retour contenant cette directive.

Cette dernière est ajoutée (coté serveur donc)

·        soit explicitement dans le code (php dans notre cas) de la page,

·        soit, dans le cas de l'utilisation d'un serveur apache, automatiquement par celui-ci. Ce mécanisme est mis en place si apache détecte dans le dossier de la page, la présence du fichier .htaccess.

Dans les exercices suivants, nous mettrons en uvre ces deux aspects.

Exercice 1 identification automatique par Apache

Voir ou revoir les parties relatives à l'authentification dans le cours http et htaccess

Que ce soit en mode automatique (htaccess) ou en mode manuel (php), les échanges entre le serveur et le client via le protocole http sont les mêmes :

1.     Envoi d'une trame de demande d'authetification www-authenticate du serveur vers le client

2.    Ouverture de la boite de dialogue permettant la saisie par l'internaute du login/mot de passe

3.    Envoi du client (navigateur) vers le serveur du login et mot de passe. Mode basic : circulation en base64 (quasi en clair) des informations d'identification.
Ces inforlations sont présente dans le champ authenticate des trames du protocole http.

En mode automatique (htaccess)

·        Le serveur prend l'initiative automatiquement d'envoyer la trame de demande d'authentification au client (navigateur)

·        Ouverture de la boite de dialogue permettant la saisie par l'internaute du login/mot de passe

·        Le serveur récupère les identifiants et les traite automatiquement : il les compare aux identifiants des utilisateurs (fichier .passwd).
Note : htacess permet des fonctionnalités plus évoluées sur la gestion des droits d'accès (ex : groupes, nom de domain)  voir poly.

En mode manuel (php)

·        Envoi explicite d'un header (instruction php) au client Le serveur envoie la trame de demande d'authentification au client (navigateur)

·        Ouverture de la boite de dialogue permettant la saisie par l'internaute du login/mot de passe

·        Le serveur récupère les identifiants et les place dans le tableau associatif $_SERVER
C'est au programmeur d'effectuer les tests pour vérifier si les identifiants sont valides (généralement dans une bdd ; on peut également utiliser un ldap).
Note : htacess permet des fonctionnalités plus évoluées sur la gestion des droits d'accès (ex : groupes, nom de domain)  voir poly.

Note1 :   La fonction setcookie() écrit des informations dans les paramètres d'entête du protocole http (à destination du client)

Note2 :  La fonction setcookie() doit être utilisée avant tout envoi de données HTML vers le navigateur, même si ces données sont envoyées avec echo, print ou équivalent (le script qui la contient doit donc être placé avant la balise <HTML> et avant toute fonction echo(),print ou printf()).

Note3 :  Le tableau $_COOKIE est initialisé automatiquement par lecture de l'entête - côté serveur. Limitation à 20 cookies.

Note4 :  Un cookie ne peut pas contenir d'information structurée. Toutefois, lors d'utilisation de tableau, il est possible que php considère les informations sous la forme d'un tableau. En réalité, dans le protocole, il y aura autant de cookie que d'élément de tableau. Il est recommandé de gérer explicitement les informations structurées (pour tous les objets  tableaux compris) en les sérialisant.

Faille de sécurité de l'identification par cookies

On souhaite pouvoir accéder à des pages protégées. On ne connaît pas le login / mot de passe permettant l'accès. Par contre, le site est construit de la même manière que précédemment.

En utilisant des outils de gestion de cookie (outils de développement du navigagteur) forcer l'accès aux pages protégées :

·        identifiez vous avec un mauvais login/mot de passe

·        à l'aide des outils de developpement web, changez la valeur du cookie "pas ok" en "ok".

·        Retentez un accès à une page protégée.
Vous devez constater que vous accedez au contenu de cette page.

Conclusion : on ne protège jamais l'accès à un site par l'utilisation directe d'un cookie car la valeur d'un cookie peut être modifiée côté client sans contrôle du serveur.

Exercice 2 : utilisation de sessions

Pour éviter les failles de sécurité d'identification par cookies, nous placerons les informations côté serveur. Pour cela, nous utiliserons des sessions. L'avantage des sessions est de stocker les informations relatives au client sur le serveur.

Lors d'une visite sur une page nécessitant une session, un identifiant unique est attribué par le serveur. Cet identifiant sera connu du client par l'utilisation d'un cookie (ou peut être propagé dans l'url si le client n'accepte pas les cookies et si la configuration du serveur le prévoit). Seule l'identifiant de session sera connu par le client. Les autres informations relatives à la session seront stockées sur le serveur.

Lire la partie cours sur les sessions.

Les principales informations de codage en php relatives aux sessions sont :

            session_start()

Ouvre la session ou la rétablit selon qu'ID de session valide est disponible. Cette fonction est à placer au début de chaque script utilisant les sessions et avant tout affichage du script.

      session_id([id])

Retourne l'ID de session en cours (si pas de paramètre d'entrée) ou force l'ID (id) de session à utiliser.

      session_destroy()

Détruit la session active. Utile si l'internaute clique que "deconnexion".

$_SESSION

tableau associatif php contenant les variables de session.

Création / initialisation d'une variable de session :

$_SESSION["ma_variable"] ="ma_valeur";

Récuperation d'une variable de session :

echo $_SESSION["ma_variable"]; // dans le flux, écrit ma_valeur.

Remarque : fermeture de session (non exhaustif).

-        Commande session_destroy()

-        Automatiquement, par le serveur après une durée (par défaut de 3 heures). Le délai peut être modifié de nb_minutes  par l'appel de session_cache_expire(nb_minutes). Dans ce cas, le décompte débute lors de l'appel de la fonction.

-        A noter : la fermeture du navigateur ne termine pas la session (sur le serveur).
